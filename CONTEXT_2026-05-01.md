- copiar el dump a `/tmp`
- dar permisos de lectura
- restaurar desde `/tmp`

Comandos relevantes usados/indicados:

```bash
cp /root/Backup_04_28.sql /tmp/Backup_04_28.sql
chmod 644 /tmp/Backup_04_28.sql
sudo -u postgres pg_restore --verbose --no-owner --role=discogs_app -d discogs_catalog /tmp/Backup_04_28.sql
```

#### Problema detectado

- en desktop, al buscar, salia `Failed to fetch`

- si no existe DSN local, entonces hacen fallback a la API remota

Resultado:

- `discogs-live` sigue local con token del usuario
- `catalog-local` usa local si hay DSN; remoto solo como fallback
- `catalog-hybrid` usa local + token local si hay DSN; remoto solo como fallback

### `discogs-live`

- sigue ejecutandose en desktop/Tauri
- usa el token personal guardado localmente por usuario
- no depende de PostgreSQL local

### `catalog-local`

- si hay DSN local guardado en desktop:
  - usa `tokio-postgres` en Tauri local
- si no hay DSN local:
  - intenta usar la API remota de catalogo por SSE

### `catalog-hybrid`

- si hay DSN local guardado en desktop:
  - usa catalogo local via Tauri
  - y detalles live con el token local del usuario
- si no hay DSN local:
  - usa la API remota

### Recordatorio pendiente MUY importante sobre token

En fallback/API remota:

- `catalog-hybrid` usa el `DISCOGS_TOKEN` del servidor/API

Esto quedo explicitamente pendiente para revisar mas adelante si se quiere:
