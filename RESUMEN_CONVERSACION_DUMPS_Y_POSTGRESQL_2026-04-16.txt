1. OBJETIVO GENERAL

Se quiere redisenar el flujo de busqueda de 103 FINDER para que sea mucho mas rapido y preciso, especialmente en busquedas complejas con muchos filtros. La idea principal es usar los dumps mensuales de Discogs para resolver localmente o en servidor la parte estructural del catalogo, y dejar la API live de Discogs solo para los datos dinamicos realmente importantes.


2. CONTEXTO DE LA APP ACTUAL

La app actual usa Discogs para buscar releases y filtrar resultados. El flujo principal de busqueda desktop se apoya en estos archivos:

- web/frontend/src-tauri/src/lib.rs
- web/frontend/lib/discogs/search-stream.ts
- web/frontend/lib/search/session.ts

El flujo actual hace:

- llamada a /database/search
- por cada candidato, llamada a details del release
- a veces fallback a main_release
- si el filtro de versiones esta activo, llamada a masters/{id}/versions

Esto hace que algunas busquedas grandes tarden mucho y puedan parecer bloqueadas o silenciosas.


3. CAMBIO YA APLICADO EN LA APP

Se modifico el watchdog de la busqueda para que ya no corte automaticamente la busqueda cuando Discogs tarda en responder. En vez de abortar, ahora muestra mensajes informativos y deja la busqueda viva hasta que llegue done o el usuario cancele.

Archivos modificados:

5. INVESTIGACION SOBRE LA API DE DISCOGS

Se reviso la documentacion publica de Discogs:

- https://www.discogs.com/developers/

Puntos importantes:

- rate limit autenticado: 60 requests por minuto por IP
- rate limit no autenticado: 25 requests por minuto por IP
- headers importantes: X-Discogs-Ratelimit, X-Discogs-Ratelimit-Used, X-Discogs-Ratelimit-Remaining
- Discogs exige User-Agent unico y claro
- existen monthly data dumps del catalogo

- versions solo deberia consultarse cuando el filtro este activado de verdad


7. PRUEBA REAL SOBRE MULTIPLES STYLES EN DISCOGS

Se hizo una prueba directa contra la API de Discogs, fuera del programa, para ver si repetir el parametro style hacia un AND real.

Pruebas:

- genre=Electronic&style=Techno&year=1991-1995
- genre=Electronic&style=Techno&style=Synth-pop&year=1991-1995
- genre=Electronic&style=Synth-pop&year=1991-1995
- genre=Electronic&style=Synth-pop&style=Techno&year=1991-1995

Resultado observado:

- Techno solo -> 54236 items
- Techno + Synth-pop -> tambien 54236 items y mismos primeros resultados Techno
- Synth-pop solo -> 49013 items
- Synth-pop + Techno -> vuelve a resultados equivalentes a Techno

Conclusion:

- no se puede confiar en que Discogs haga AND real al mandar multiples parametros style
- repetir style no garantiza precision
- por eso la app debe validar localmente el AND de estilos si se quiere exactitud


8. IDEA PRINCIPAL A DESARROLLAR

La idea estrategica principal es:

- descargar los monthly data dumps de Discogs
- importarlos en una base propia
- hacer el filtrado estructural en esa base propia
- obtener una lista corta y precisa de IDs candidatos
- solo despues consultar Discogs live para datos dinamicos

Ventajas:

- AND real de estilos y generos
- mucho menos trafico a Discogs
- menos riesgo de rate limit
- mejor velocidad en busquedas complejas
- control total sobre filtros estructurales

10. DUMPS QUE HAY QUE DESCARGAR

No hace falta descargar historicos por ano. Un dump actual ya es un snapshot completo del catalogo disponible en Discogs en ese momento, incluyendo releases antiguos y modernos.

Los 4 dumps recomendados son:

- discogs_YYYYMMDD_releases.xml.gz
- discogs_YYYYMMDD_masters.xml.gz
- discogs_YYYYMMDD_artists.xml.gz
- discogs_YYYYMMDD_labels.xml.gz

Y tambien:

- discogs_YYYYMMDD_CHECKSUM.txt

Prioridad:

1. releases
2. masters
3. artists
4. labels

13. RESPUESTA A SI BASTA EL DUMP DE ESTE MES

Si. Para el catalogo historico actual de Discogs, basta con el dump actual. No hace falta descargar un dump por cada ano. El dump actual incluye material de 1978 hasta la actualidad y tambien anterior, siempre que este presente en Discogs.

Lo que no sustituye el dump:

- have / want actuales
- precios y copias en venta
- marketplace
- datos de usuario
- cambios futuros hasta el siguiente dump mensual


14. PROPUESTA DE MODELO DE DATOS

En vez de guardar listas gigantes tipo "todos los IDs techno", se propuso una BD indexada con tablas relacionales.

Tablas principales:

- releases
- masters
- artists
- labels

Tablas relacionales:

- release_styles
- release_genres
- release_formats
- release_artists
- release_labels

Tablas opcionales utiles:

- release_tracklist
- release_identifiers
- release_companies

Ejemplo de AND real de estilos en SQL:

select release_id
from release_styles
where style in ('Techno', 'Synth-pop')
group by release_id
having count(distinct style) = 2;


15. COMO RESOLVER MAX_VERSIONS CON LOS DUMPS

Se concluyo que max_versions es uno de los mejores filtros para mover al dump.

Idea correcta:

- no pedir masters/{id}/versions a Discogs cada vez
- usar la relacion release -> master_id del dump
- contar cuantas releases pertenecen a cada master

Tabla recomendada:

- master_version_counts(master_id, version_count)

Ejemplo de calculo:

create table master_version_counts as
select master_id, count(*) as version_count
from releases
where master_id is not null
group by master_id;

Con eso, max_versions se puede filtrar localmente sin requests live.


16. ARQUITECTURA RECOMENDADA

Arquitectura ideal:

- Base propia/PostgreSQL: catalogo Discogs estructural
- Discogs API live: have, want, precio, copias, collection/wantlist y datos dinamicos
- Supabase: auth, usuarios, favoritos, metricas, billing y datos propios de la app
- Desktop app: cliente principal con token local del usuario

Flujo ideal:

1. la app consulta primero la base propia para filtrar estilos/generos/ano/pais/formato/versions/etc.
2. obtiene una lista corta de IDs candidatos
3. luego llama a Discogs live solo para have, want, precio y copias
4. devuelve resultados finales


17. ESTIMACION DE MEJORA DE RENDIMIENTO

Para busquedas muy complejas con casi todos los filtros activos, se estimo una mejora realista aproximada de:

- 70% a 90% mas rapida

Motivo principal:

- ahora la app puede llegar a hacer cientos de details y muchas calls de versions
- con la base propia, se filtran la mayoria de candidatos antes de tocar Discogs live
